# Harness Engineering 第二部分：长程自循环理论

> 长程自循环 = 无状态会话 + 有状态文件系统 + 固定启动/退出协议。

---

## 1. 根本事实

AI 会话是一次性的，但项目不是。

单次会话能做 1-3 个任务就到上下文上限。长程工作需要把"人的接续能力"用工程手段补上——把状态全部沉淀到文件，加上固定的启动序列和退出协议，用工程手段把"会断的会话"拼成"不断的工作流"。

---

## 2. 四个基石

### 基石 1：状态持久化

- **问题**：会话间断记忆丢，新会话不知道之前做到哪
- **解法**：追加式 `progress.md`，记录每个任务的 START/DONE/FAIL/SESSION END
- **案例**：Anthropic 的 `claude-progress.txt`

### 基石 2：自验证

- **问题**：AI 乐观报告"做完了"，没有客观校验
- **解法**：验证脚本读任务验收标准，逐条 PASS/FAIL
- **关键**：验证基准来自 PRD，不是来自 AI 自己写的代码
- **案例**：LangChain 的 Verify 阶段对照原始需求

### 基石 3：自复盘

- **问题**：同样的错反复犯
- **解法**：追加式 `lessons.md`，每次犯错记录现象+归因+修复+新规则
- **案例**：Mitchell Hashimoto 的"免疫系统"——文档随错误进化

### 基石 4：中断恢复

- **问题**：异常中断后状态混乱
- **解法**：新会话做完整性检查——读 progress.md 最后一条，对照实际代码判断是否真完成

---

## 3. 三层结构

```
Driver（调度）→ 启动会话 → Session（干活）→ 读写 → State（记忆）
     ↑                                              │
     └──── 读 progress.md 决定是否再启动 ←───────────┘
```

| 层级 | 角色 | 职责 |
|------|------|------|
| 外层 Driver | 调度 | 读 progress.md，有任务且无阻塞就启动新会话 |
| 中层 Session | 干活 | 按启动序列恢复上下文 → 取任务 → 做 → 验证 → 复盘 → 写进度 |
| 底层 State | 记忆 | progress.md / lessons.md / tasks/active/ / tasks/completed/ |

**为什么分三层**：AI 会话本身是无状态、会中断的。把"调度"和"记忆"分别放到外层和底层，中层会话就变成了纯粹的"读输入→产出→写输出"的无状态函数，可以任意重启而不丢进度。

---

## 4. 单次会话启动序列（8 步，不得跳过）

1. pwd 确认工作目录
2. 读 AGENTS.md（行为准则）
3. 读 progress.md（做到哪了）
4. 读 lessons.md（别重犯什么错）
5. git log --oneline -5（最近代码变了什么）
6. 读 tasks/active/（下一个任务）
7. smoke test（环境还正常吗）
8. 然后才开始做新任务

---

## 5. 四个停止条件

| 条件 | 动作 |
|------|------|
| active/ 任务清空 | 写 ALL DONE，退出 |
| 连续 2 个任务失败 | 写 BLOCKED，退出 |
| 会话 turn 接近上限 | 写 SESSION END + next，退出 |
| 需要人决策 | 写 WAITING DECISION，退出 |

---

## 6. 本质

把记忆从脑子里搬到纸上，让流程不依赖任何单个会话的存活。这跟数据库的事务日志、CI/CD 的 pipeline 状态机、人类公司的交接文档，是同一个思想。
