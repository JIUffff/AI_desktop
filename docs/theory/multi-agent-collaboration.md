# Harness Engineering 第三部分：多代理协作理论

> 多代理协作 = 把复杂任务拆成有界职责的子任务，分配给独立子代理并行执行，再由主代理整合。本质是分布式系统思想在 AI 工程上的应用。

---

## 1. 为什么需要多代理

单代理的三大问题：
1. **上下文爆炸**——一个代理包揽所有子任务，上下文被无关信息填满
2. **职责模糊**——什么都管等于什么都管不好
3. **无法并行**——串行执行，进度被最慢环节拖死

案例：OpenAI 维护 88 个 AGENTS.md 文件，每个子系统一个。

---

## 2. 任务分解机制

### 分解三原则

| 原则 | 含义 | 违反的代价 |
|------|------|-----------|
| MECE | 相互独立、完全穷尽 | 重叠→冲突；遗漏→做不完 |
| 原子性 | 可被一个子代理独立完成 | 频繁跨代理沟通→通信成本吞没并行收益 |
| 可验证 | 有客观验收标准 | 无法验证→主代理无法判断谁错了 |

### 分解四步法

```
原始任务 → ① 按领域/技能拆分 → ② 按依赖排序 → ③ 定义输入输出契约 → ④ 输出 DAG
```

---

## 3. 子代理角色分配

每个子代理必须明确三要素：

| 要素 | 含义 |
|------|------|
| 职责 | 要完成什么 |
| 能力边界 | 不能做什么（Non-Goals） |
| 上下文范围 | 能读哪些文件、能改哪些代码 |

### 类型匹配

| 子任务特点 | 子代理类型 |
|-----------|-----------|
| 深度推理（架构设计） | reasoning |
| 高频简单操作 | lite |
| 读取大量代码理解 | Explore（只读） |
| 写文件、跑命令 | general-purpose |

### 上下文隔离

每个子代理有独立上下文窗口。主代理只传递子任务所需的最小上下文，避免上下文爆炸和越界。

---

## 4. 协作与通信方式

### 三种通信模式

| 模式 | 机制 | 适用场景 |
|------|------|---------|
| 共享黑板 | 读写同一共享文件 | 子任务独立性强 |
| 消息传递 | 主代理转交上游产出 | 子任务有依赖 |
| 任务队列 | 主代理维护队列 | 并行同质任务 |

### 核心规则

**子代理之间不直接通信**。所有跨子代理的信息流经由主代理中转，防止"闲话式同步"产生隐性依赖。

### 共享上下文载体

```
docs/spec/*.md        ← 技术设计，所有代理可读
docs/tasks/active/    ← 任务状态，主代理维护
src/                  ← 代码仓库，按职责边界修改
shared_context.json   ← 结构化上下文（接口定义、数据格式）
```

---

## 5. 结果整合

### 三步流程

```
① 收集 → ② 验证（对照 PRD 的 AC）→ ③ 整合（按架构拼接 + 集成测试）
```

### 验证基准

**主代理验证子代理的基准来自 PRD，不是来自主代理自己的偏好。**

案例：LangChain Verify 阶段对照原始需求，不是看代码是否"看起来对"。

### 整合失败处理

| 失败类型 | 处理 |
|---------|------|
| 子代理产出不符合 AC | 记录原因 → 重派子任务 |
| 集成测试失败 | 定位冲突模块 → 协调接口 |
| 根本性矛盾 | 升级人工介入 |

---

## 6. 冲突处理

### 四种冲突类型

| 类型 | 示例 | 危害 |
|------|------|------|
| 输出矛盾 | 接口返回类型不一致 | 中 |
| 依赖冲突 | B 等 A 的产出但 A 没完成 | 高 |
| 资源冲突 | 两个代理同时改一个文件 | 高 |
| 设计冲突 | A 用 PyTorch，B 用 TensorFlow | 中 |

### 五种协调策略

1. **预防为主**：Spec 阶段锁定接口契约
2. **主代理仲裁**：读 PRD 裁决哪个符合需求
3. **依赖排序**：严格按 DAG 拓扑排序
4. **文件锁**：子代理获取文件锁后才能写
5. **升级机制**：无法自动解决时升级人工介入

---

## 7. 与 Harness Engineering 的关系

| Harness Engineering 环节 | 多代理协作中的对应 |
|--------------------------|------------------|
| PRD | 主代理分解任务的依据，AC 是验证基准 |
| Spec | 子代理间的接口契约，预防冲突的关键 |
| Tasks | 子任务 DAG，每个任务对应一个子代理 |
| Implementation | 子代理并行执行阶段 |
| Validation | 主代理结果整合与验证 |
| Review | 冲突处理与升级机制 |

**结论**：多代理协作是 Harness Engineering 在复杂任务场景下的自然延伸。没有 PRD/Spec/Tasks 文档驱动，多代理协作就是空中楼阁。
