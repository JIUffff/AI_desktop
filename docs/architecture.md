## 四层架构（UI-TARS 端到端版）

```
┌──────────────────────────────────────────────────────────┐
│                    用户交互层                              │
│        自然语言输入  ·  进度反馈  ·  人工确认 UI           │
└────────────────────────────┬─────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────┐
│         决策感知层 — UI-TARS 原生 Agent                     │
│    ┌──────────────────────────────────────────┐          │
│    │  UI-TARS-1.5-7B（AWQ INT4）              │          │
│    │  内置能力：OCR + 元素定位 + 推理 + 动作     │          │
│    │  输出：Thought + Action（0-1000 坐标）      │          │
│    │  显存：~8GB（含 ViT 视觉编码器）            │          │
│    │  备选：Qwen3-VL-8B (GPTQ-Int4, ~5GB)      │          │
│    └──────────────────────────────────────────┘          │
│         ↕ ui-tars action parser 解析输出                    │
└────────────────────────────┬─────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────┐
│              安全控制层（增强版）                           │
│    风险评级 · 动作校验 · 循环检测 · 人工确认               │
│    审计日志 · 预算熔断 · 输出格式校验                      │
└────────────────────────────┬─────────────────────────────┘
                             │
┌────────────────────────────▼─────────────────────────────┐
│              执行层                                       │
│    PyAutoGUI（鼠标键盘）· Pywinauto（Win 控件）          │
│    Playwright（浏览器）· 文件系统 API                    │
└──────────────────────────────────────────────────────────┘

[可选] YOLO11m 插件 — 按需 lazy-load，不参与主流程
```

### 与旧架构的关键区别

| 维度 | 旧架构（通用 VLM） | 新架构（UI-TARS Agent） |
|------|-------------------|------------------------|
| **层数** | 5 层（用户→决策→视觉→安全→执行） | 4 层（用户→决策感知→安全→执行） |
| **模型数** | 4 个常驻模型（VLM+YOLO+OCR+ViT） | 1 个常驻模型（UI-TARS 含 ViT） |
| **感知方式** | 三重检测流水线（YOLO→OCR→SoM） | 端到端原生能力（内置 OCR+检测+推理） |
| **循环步数** | 7 步 / 次 | 4 步 / 次 |
| **峰值显存** | ~14GB | **~10-11GB** |
| **延迟预期** | ~800-1500ms | **~320-500ms** |
| **误差来源** | 多模型串联累积误差 | 单模型端到端，无串联误差 |

## OODA 运行时序（简化版）

```
用户输入指令
    ↓
[决策感知层] 解析意图 → 分解为 N 步任务
    ↓
[循环开始]
    ↓
[截图] 截取当前屏幕 → 传入 UI-TARS
    ↓
[UI-TARS 推理] 截图 + 当前任务步骤 → Thought + Action
    ├── 内置 OCR：自动识别屏幕文字
    ├── 内置 Grounding：定位目标元素坐标 (0-1000)
    └── 内置 Reasoning：决定动作类型和参数
    ↓
[Action Parser] 解析文本输出 → 结构化指令 {action, coordinates, params}
    ↓
[安全层] 校验：
    ├── 坐标范围 [0, 1000]？
    ├── 动作类型白名单？
    ├── 风险等级 → 是否需要人工确认？
    └── 循环检测 → 是否卡死在重复操作？
    ↓
[执行层] 执行动作（PyAutoGUI / Pywinauto / Playwright）
    ↓
[再次截图] → [UI-TARS 验证] → 任务是否完成？
    ↓
[循环结束] → 任务完成 → 通知用户 + 写审计日志
```

## 显存分配

### 主方案：UI-TARS-1.5-7B AWQ INT4（推荐）

| 模块 | 显存 | 说明 | 变更 |
|------|------|------|------|
| UI-TARS-1.5-7B (AWQ INT4) | ~6GB | 主 Agent（含 LLM backbone） | — |
| ViT 视觉编码器 | ~2GB | 随主模型加载，内置 OCR + 元素检测 | — |
| KV Cache | ~3~4GB | 推理缓存（因移除其他模型可扩容） | ⬆️ 从 2GB 扩容 |
| 图像缓冲（截图） | ~1GB | 截图/历史帧 | — |
| **总峰值** | **~10~11GB** | | **⬅️ 从 14GB 降低！** |
| **余量** | **~5~6GB** | RTX 5070 Ti 16GB | **✅ 大幅改善** |

### 备选方案：Qwen3-VL-8B GPTQ-Int4（通用场景）

| 模块 | 显存 | 说明 |
|------|------|------|
| Qwen3-VL-8B (GPTQ-Int4) | ~3.1GB | 通用 VLM（极度省显存） |
| ViT 视觉编码器 | ~2GB | 随主模型加载 |
| KV Cache | ~4~5GB | 推理缓存（256K 上下文支持） |
| 图像缓冲 | ~1GB | 截图/历史帧 |
| **总峰值** | **~10~11GB** | **留 5~6GB 余量** |

### 可选插件（按需加载）

| 插件 | 显存 | 加载策略 | 用途 |
|------|------|----------|------|
| YOLO11m | ~2GB | Lazy-load，用完卸载 | 超高速纯检测场景（非主流程） |
| PaddleOCR v4 | ~1GB | Lazy-load，仅 fallback | VLM OCR 失败时的降级方案 |

## 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 视觉驱动 vs 控件树 | 视觉驱动为主，控件树为辅 | 通用性强，能操作任意 GUI |
| 云端 vs 本地 | 全本地 | 隐私、成本、可定制 |
| GUI Agent vs 通用 VLM | **GUI Agent（UI-TARS）** | ScreenSpot-V2 94.2% SOTA，端到端原生能力，无需辅助 CV |
| 端到端 vs 流水线 | **端到端** | 消除多模型串联误差，降低 60% 延迟，节省 3-4GB 显存 |
| FP16 vs INT4 | INT4 量化（AWQ/GPTQ） | 显存预算决定，精度损失 <1% |
| 一次一个 vs 批量 | 一次一个任务 | Harness Engineering 规范 |
| 安全防线位置 | Action Parser 之后、执行之前 | UI-TARS 直接输出可执行动作，安全层是最后且唯一的防线 |
